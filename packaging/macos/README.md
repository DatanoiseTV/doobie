# macOS release signing + notarization

Doobie's macOS releases are built, code-signed with a Developer ID
Application certificate, packaged as a signed `.pkg` installer, notarized
through Apple's notary service, and stapled — so end users install
without a Gatekeeper warning. The whole pipeline runs in CI from a clean
GitHub Actions runner; the only thing stored locally is what's needed
once for the initial bootstrap.

This directory holds everything that pipeline depends on.

## Files

| Path | What it is |
| --- | --- |
| `entitlements.plist` | Hardened-runtime entitlements applied to every bundle (AU/VST3/Standalone). The three exceptions an audio plug-in actually needs and nothing more. |
| `distribution.xml`  | productbuild distribution descriptor that combines the three component pkgs into a single user-facing installer. Versions are substituted at build time. |
| `welcome.txt`       | The first screen of the installer. |
| `build-pkg.sh`      | The actual sign → bundle → notarize → staple pipeline. Called by `release.yml` after CMake builds the artefacts; runnable locally for debugging too. |
| `bootstrap-signing.sh` | One-shot script that brings a fresh Apple Developer account from "nothing set up" to "all 11 GitHub Secrets populated". Handles the Apple paperwork as far as Apple's API actually lets us; gracefully falls back to opening the portal for the bits Apple's API refuses to do. |
| `setup-secrets.sh`  | Lower-level helper. If you already have your two `.p12` files + the App Store Connect `.p8` on disk, this just sets the GitHub Secrets without going through cert creation. |

## The 11 GitHub Secrets

`release.yml` reads these. Set them once with `bootstrap-signing.sh`
(or `setup-secrets.sh` if you brought your own files):

| Secret | Source | Notes |
| --- | --- | --- |
| `MACOS_CERTIFICATE`                  | Developer ID **Application** `.p12`, base64-encoded   | Bytes streamed via stdin so the value never appears in shell history. |
| `MACOS_CERTIFICATE_PWD`              | Password for that `.p12`                              | Random 32-hex if generated by `bootstrap-signing.sh`. |
| `MACOS_INSTALLER_CERTIFICATE`        | Developer ID **Installer** `.p12`, base64-encoded     | Same handling. |
| `MACOS_INSTALLER_CERTIFICATE_PWD`    | Password for that `.p12`                              |  |
| `APPLE_NOTARY_KEY_P8`                | App Store Connect API key `.p8`, base64-encoded       | Created at appstoreconnect.apple.com → Users and Access → Integrations → Keys. **One-time download** — keep a backup. |
| `APPLE_NOTARY_KEY_ID`                | Key ID (10-char alphanumeric)                         | Visible in the Keys table on the page above; also the suffix in the `.p8` filename (`AuthKey_<KEYID>.p8`). |
| `APPLE_NOTARY_ISSUER_ID`             | Issuer ID (UUID)                                      | Shown above the Keys table on the same page. |
| `APPLE_TEAM_ID`                      | Team ID (10-char)                                     | At developer.apple.com → Membership. |
| `MACOS_APPLICATION_IDENTITY`         | Subject CN of the Application cert                    | e.g. `Developer ID Application: Your Name (TEAMID)`. Derived automatically by `bootstrap-signing.sh` from the `.cer`. |
| `MACOS_INSTALLER_IDENTITY`           | Subject CN of the Installer cert                      | e.g. `Developer ID Installer: Your Name (TEAMID)`. |
| `MACOS_KEYCHAIN_PWD`                 | Anything — used to unlock the ephemeral CI keychain   | Random hex generated by the bootstrap script; not exposed elsewhere. |

## What Apple's API actually lets us do (state as of 2026-05)

The bootstrap script tries the App Store Connect API for as much as
possible and only opens the web portal for the bits Apple still refuses
to expose. Reality check:

* **App Store Connect API key creation**: not exposed by any API. One-time
  manual visit to the Integrations page is unavoidable (chicken and egg
  — you need the key to call the API).
* **Developer ID Application cert (`DEVELOPER_ID_APPLICATION_G2`)**: the
  enum *is* in the API, but `POST /v1/certificates` rejects it with
  `FORBIDDEN_ERROR` unless the API key was issued with the **Account
  Holder** role. The Admin role is one rung short. If you regenerate the
  ASC API key with Account Holder scope you can automate this; otherwise
  the bootstrap falls back to uploading the CSR it generated to the web
  portal (5 clicks).
* **Developer ID Installer cert**: `DEVELOPER_ID_INSTALLER` is not in
  Apple's API at all. The API's 409 error message confirms the supported
  certificate types and `DEVELOPER_ID_INSTALLER` simply isn't on the
  list. Portal-only.
* **Everything past cert issuance**: fully automatable — CSR generation,
  `.cer → .p12` bundling with random passwords, keychain bootstrap on the
  runner, `codesign` + `productbuild` + `productsign`, `xcrun notarytool
  submit --wait`, `stapler`. None of this needs a human.

## Triggering a release

* **Nightly**: every push to `main` updates the rolling `nightly`
  GitHub release with fresh signed + notarized artefacts.
* **Versioned**: push a tag matching `v*` (e.g. `git tag -a v0.13.0 -m
  ...` then `git push origin v0.13.0`). The workflow publishes a
  stable release tagged the same way.
* **Manual**: from the Actions tab, "Run workflow" on `release.yml`.
  Produces a one-off `manual-<timestamp>-<sha>` prerelease.

## Rotating credentials

If a cert is compromised or expires:

1. Revoke the old cert at developer.apple.com if needed.
2. Re-run `bootstrap-signing.sh`. It generates a fresh private key + CSR,
   issues a new cert, bundles it, and overwrites the relevant GitHub
   Secrets. The old `.p12` becomes unused.

The App Store Connect `.p8` cannot be re-downloaded if lost — you
revoke and re-issue. Same script will pick up the new one.
